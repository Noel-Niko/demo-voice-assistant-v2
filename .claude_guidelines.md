# Development Guidelines for This Project

## Git & Repository Rules
- **NEVER push to the remote repository** - All pushes will be done manually by the user
- Only commit when explicitly requested
- Always create meaningful commit messages when commits are requested

## Code Quality Principles

### SOLID Principles
- **Single Responsibility**: Each class/function should have one clear purpose
- **Open/Closed**: Open for extension, closed for modification
- **Liskov Substitution**: Subtypes must be substitutable for their base types
- **Interface Segregation**: Many specific interfaces are better than one general interface
- **Dependency Inversion**: Depend on abstractions, not concretions

### DRY (Don't Repeat Yourself)
- Eliminate code duplication
- Extract common logic into reusable functions/classes
- Use inheritance and composition appropriately

### Clean Code
- Write self-documenting code with clear naming
- Keep functions small and focused
- Maintain consistent formatting and style
- Remove dead code and unused imports
- Prefer readability over cleverness

## Complexity and Solutions
- **ALWAYS prefer simple, reliable solutions over complex ones**
- Avoid over-engineering - implement only what's needed
- If a solution feels too complex, ask for clarification or propose a simpler alternative
- Straightforward code is better than "clever" code
- Three clear lines are better than one cryptic line

## Communication and Assumptions
- **ALWAYS ask before making assumptions**
- When requirements are unclear, ask for more information
- **Never guess** - request clarification instead
- If multiple approaches are possible, present options and ask which to use
- Confirm understanding before implementing significant changes

## Testing Requirements
- **ALWAYS write unit tests to verify methods work correctly**
- Test individual methods in isolation
- Test integration between methods and components
- Run tests after implementation to verify everything works
- Include both positive and negative test cases
- Ensure tests are readable and maintainable

## Summary Checklist
Before completing any task, verify:
- [ ] Did I ask for clarification instead of guessing?
- [ ] Is this the simplest solution that works?
- [ ] Does it follow SOLID and DRY principles?
- [ ] Is the code clean and readable?
- [ ] Have I written unit tests?
- [ ] Have I tested integration between components?
- [ ] Did I avoid pushing to the remote repository?
